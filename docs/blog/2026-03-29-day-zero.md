# cc-praxis — Day Zero: The First Nine Skills

**Date:** 2026-03-29
**Type:** day-zero

---

## What I Was Trying To Achieve

I've been using Claude Code for a while, and I kept noticing the same friction: every new session started cold. I'd re-explain my conventions, re-establish context, re-teach things Claude had already learned three conversations ago. The knowledge was ephemeral — useful in the moment, gone immediately after.

I wanted a way to encode the things I'd figured out — about Java, Quarkus, how I like commits structured, how I think about code review — so Claude would just *know* them at the start of every session. Not re-learn. Not be told. Know.

Claude Code has a skills mechanism for exactly this. Skills are Markdown files with YAML frontmatter describing when to load them. When the trigger condition fires, the skill loads automatically and Claude has the full workflow. That's the idea.

## What I Believed Going In

I believed this was mostly a writing problem. I knew what I wanted Claude to do in each context — commit messages, code review, dependency updates, observability. I just needed to write those workflows down in a format Claude could load and follow. Simple enough.

I also believed the main challenge would be getting the *descriptions* right — the YAML frontmatter that determines when a skill triggers. Too vague, and the skill loads constantly and wastes context. Too narrow, and it never loads when you need it.

What I didn't fully appreciate at this point was how much discipline the *structure* of the skills themselves would require — especially around cross-referencing, consistency, and preventing the skills from drifting apart over time. That would become a much bigger concern than I expected.

## What We Built

In the first commit — just nine skills — we focused exclusively on what I use every day with Java and Quarkus:

- `java-dev` — the core development workflow, all the conventions I've built up
- `java-git-commit` — structured commits with DESIGN.md sync
- `adr` — Architecture Decision Records in MADR format
- `quarkus-flow-dev` — Quarkus Flow specific patterns (this one was large, ~350 lines)
- `update-design` — keeping DESIGN.md in sync with code changes
- `code-review` — code review checklist
- `dependency-update` — Maven BOM management
- `logging-observability` — the full observability setup we use
- A basic `.gitignore`

Nine skills, 1,625 lines of Markdown, committed in a single shot.

The `quarkus-flow-dev` skill was the longest and most technical — it encoded a lot of hard-won knowledge about the FuncDSL API, workflow patterns, and testing approaches. That skill alone made the whole project worthwhile; it captured things that had taken months to figure out.

## What Changed Immediately

Within the next few days, almost everything about the *structure* of the skills changed, even if the underlying content didn't. I hadn't been rigorous about:

- **CSO (Claude Search Optimization)** — the description field must describe *when to trigger*, not *what the skill does*. If the description summarises the workflow, Claude might follow the description instead of reading the full skill.
- **Cross-references** — skills that invoke each other need to say so explicitly in both directions. I'd written them mostly in isolation.
- **Naming conventions** — the original names were generic (`code-review`, `dependency-update`). I renamed them to be language-prefixed where appropriate and created generic principle skills for the rest.
- **Mermaid flowcharts** — I'd initially used GraphViz dot notation, which turned out not to be the right format. We converted everything to Mermaid.

These weren't mistakes exactly — they were things I hadn't known to think about before starting. The skills worked. They just weren't as good as they could be.

## What I Now Believe

Writing skills is harder than it looks, and the hardest part isn't the content — it's the discipline. Every skill needs:
- A description that triggers at the right moment (not too broad, not too narrow)
- Cross-references that actually match what the other skills say
- A consistent structure so you can scan them all and understand how they fit together

The content — the actual Java conventions, the commit format, the code review checklist — that part was mostly already in my head. Getting it into a format that Claude loads reliably at the right moment, with everything pointing to everything else correctly, that's the real work.

I also didn't anticipate how quickly the collection would grow. Nine skills felt comprehensive at the time. Within a week it would be over forty.
